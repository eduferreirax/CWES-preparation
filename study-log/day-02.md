## Topics Covered

Today I studied the **Pre-Engagement phase** of a penetration test.

This module focuses on the legal, ethical, and procedural aspects that must be defined **before any testing begins**.

## Legal Agreements

One of the most important parts of a penetration test is defining the **legal agreements and authorization** required to perform the assessment.

### NDA (Non-Disclosure Agreement)

An NDA ensures that any sensitive information discovered during the engagement remains confidential.

Two common types are:

- **Bilateral NDA** – both parties agree to protect each other's confidential information.
- **Multilateral NDA** – used when multiple parties are involved and all agree to protect shared information.

These agreements protect both the **client and the penetration tester**.

## Authorization and Responsible Contacts

A penetration test must only be authorized by **people with the proper authority inside the organization**, such as:

- CEO
- CTO
- CISO
- Legal department
- Security leadership

Accepting testing requests from unauthorized employees could create serious legal problems.

For example, an internal employee could request a pentest without proper authorization and later claim the activity caused damage.

Because of this, several documents and approvals must be defined before testing begins, including:

- Scope definition
- Rules of engagement
- Authorization to test
- Emergency contact information

## Handling Critical Vulnerabilities

If a **critical vulnerability** is discovered during testing that could cause immediate harm to the organization, the tester should:

1. Stop the testing activity related to that issue
2. Immediately notify the authorized contact
3. Follow the agreed communication process

This ensures the issue can be handled responsibly and safely.

## Types of Penetration Testing

### Black Box Testing

The tester has **little to no internal knowledge** about the target.

The organization may only provide:

- IP addresses
- Domains
- Public-facing assets

This simulates an **external attacker** perspective.

### Grey Box Testing

The tester receives **limited internal information**, such as:

- User credentials
- Internal endpoints
- Partial infrastructure knowledge

This approach simulates an attacker who already has **some level of access** to the system.

### White Box Testing

The tester has **full access** to internal information, including:

- Source code
- Architecture documentation
- Internal infrastructure
- Application logic

This approach allows for deeper security analysis such as:

- Code review
- Logic testing
- architectural security evaluation

## Key Lesson

The pre-engagement phase is critical because it defines:

- legal protection
- scope of testing
- communication procedures
- responsibilities of both parties

Without proper preparation and authorization, a penetration test could create **legal, ethical, and operational risks**.
